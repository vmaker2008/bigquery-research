## 아키텍처 핵심 원칙

BigQuery는 **컴퓨팅과 스토리지를 완전히 분리**하는 혁신적인 아키텍처를 채택합니다.
이 설계 덕분에 각 계층이 독립적으로 확장되고 최적화될 수 있습니다.

```
┌──────────────────────────────────────────────────────┐
│                   사용자 / 애플리케이션                  │
└───────────────────────┬──────────────────────────────┘
                        │ SQL 쿼리
┌───────────────────────▼──────────────────────────────┐
│              Dremel (쿼리 실행 엔진)                    │
│  ┌──────────┐  ┌──────────┐  ┌──────────────────────┐ │
│  │ 루트 노드 │  │ 중간 노드 │  │    리프 노드 (Slots)   │ │
│  │ (조정)   │  │ (분산)   │  │    (데이터 처리)       │ │
│  └──────────┘  └──────────┘  └──────────────────────┘ │
└───────────────────────┬──────────────────────────────┘
                        │ Jupiter 고속 네트워크
┌───────────────────────▼──────────────────────────────┐
│           Colossus (분산 스토리지)                      │
│     열 기반(Columnar) 형식으로 데이터 저장               │
│     자동 복제 + 압축 + 암호화                           │
└──────────────────────────────────────────────────────┘
```

---

## 핵심 컴포넌트

### 1. Dremel — 쿼리 실행 엔진

BigQuery의 두뇌 역할을 하는 분산 쿼리 처리 엔진입니다.

#### 트리 구조 아키텍처

```
              [루트 노드]
           쿼리 수신 및 조정
          /               \
    [중간 노드]          [중간 노드]
    쿼리 분배             쿼리 분배
    /     \              /     \
[리프] [리프]         [리프] [리프]
실제 데이터 스캔 및 처리
```

#### 주요 특징
- **병렬 처리**: 수천 개의 머신에 동시에 쿼리 실행 분산
- **컬럼 형식**: 분석 쿼리에 최적화된 열 기반 데이터 처리
- **Slot**: 쿼리 실행의 기본 단위 (CPU, 메모리, 네트워크 묶음)
- **성능**: 테라바이트 규모 스캔을 초 단위로 완료

---

### 2. Colossus — 분산 스토리지

Google의 차세대 분산 파일 시스템으로 BigQuery의 데이터 저장소 역할을 합니다.

#### 특징

| 기능 | 설명 |
|------|------|
| **열 기반 저장** | 분석 쿼리 시 필요한 열만 읽어 성능 향상 |
| **자동 복제** | 여러 데이터센터에 자동 복제하여 고가용성 보장 |
| **압축** | 고급 압축으로 스토리지 비용 절감 |
| **암호화** | 저장 데이터 자동 암호화 |
| **자동 복구** | 디스크 장애 시 자동 복구 |
| **무한 확장** | 스토리지 용량 제한 없음 |

---

### 3. Jupiter — 고속 내부 네트워크

Google 데이터센터 내부의 초고속 네트워킹 인프라입니다.

#### 역할
- **컴퓨팅 ↔ 스토리지** 간 초고속 데이터 전송
- **페타비트/초** 규모의 총 양분할 대역폭
- 대규모 분산 **JOIN**과 **집계** 연산 지원
- 기본 중복성으로 신뢰성 있는 데이터 전송 보장

> **핵심**: Jupiter 덕분에 스토리지와 컴퓨팅이 분리되어 있어도 마치 로컬에서 처리하는 것처럼 빠른 성능이 가능합니다.

---

### 4. Borg — 클러스터 오케스트레이션

Kubernetes의 선구자인 Google의 클러스터 관리 시스템입니다.

- Dremel 슬롯에 하드웨어 자원 할당
- Google 글로벌 인프라에서 컴퓨팅 자원 최적 배분
- 장애 허용성 및 자동 복구 제공

## 컴퓨팅과 스토리지 분리의 장점

```
전통적 아키텍처 (결합형)          BigQuery 아키텍처 (분리형)
━━━━━━━━━━━━━━━━━━━━━━━━         ━━━━━━━━━━━━━━━━━━━━━━━━━
컴퓨팅 ↔ 스토리지 결합            컴퓨팅 | 스토리지 독립
확장 시 모두 함께 확장             필요한 것만 독립 확장
데이터 이동 = 병목                 Jupiter로 초고속 전송
고정 비용 구조                     탄력적 비용 구조
```

## BigQuery 슬롯 (Slots)

**슬롯(Slot)**은 BigQuery 쿼리를 실행하는 가상 CPU 단위입니다.

- 각 슬롯은 CPU, 메모리, 네트워크 자원을 포함
- 복잡한 쿼리일수록 더 많은 슬롯이 필요
- **온디맨드**: Google이 자동으로 슬롯 관리
- **예약(Reservation)**: 사용자가 슬롯 수를 미리 예약

```
슬롯 예시:
쿼리 A → 1,000 슬롯 병렬 사용 → 빠른 결과
쿼리 B → 500 슬롯 병렬 사용  → 그에 맞는 속도
```
